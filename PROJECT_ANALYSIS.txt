1. PROJECT OVERVIEW
================================================================================

BurnerDrop is a zero-trust, client-side encrypted file sharing web application.
It allows anyone to securely share files without accounts, without login, and
without any server ever seeing the file contents.

Core Philosophy: "Your browser is the vault."

All encryption and decryption happens inside the browser using the native
Web Crypto API. The server (IPFS) only stores unintelligible encrypted blobs.
Even the BurnerDrop service itself cannot read user files.

Key Value Proposition:
  - No accounts required
  - No tracking, no logs, no analytics
  - End-to-end encrypted (AES-256-GCM)
  - Decentralized storage (IPFS)
  - Dual credential system (CID + Password) for enhanced security
  - Completely open source
  - Works in any modern browser

2. HOW IT WORKS (TECHNICAL FLOW)
================================================================================

SENDING A FILE:
---------------
  Step 1: User drops a file into the browser (max 10 MB)
  Step 2: Browser generates a fresh AES-256-GCM key (via crypto.subtle)
  Step 3: Browser generates a random 96-bit IV (Initialization Vector)
  Step 4: File is encrypted entirely client-side (plaintext never leaves device)
  Step 5: Metadata (filename, type) + IV are bundled into the encrypted blob
  Step 6: Encrypted blob is uploaded to IPFS via Pinata API
  Step 7: User receives TWO credentials:
          - CID  (Content Identifier — where the file lives on IPFS)
          - KEY  (AES-256 decryption password in human-readable format)

RECEIVING A FILE:
-----------------
  Step 1: Receiver enters both CID and Password
  Step 2: Encrypted blob is fetched from IPFS using the CID
  Step 3: IV and metadata are extracted from the blob header
  Step 4: Password is converted back to an AES-256 CryptoKey
  Step 5: File is decrypted using AES-GCM (if key is wrong, it fails hard)
  Step 6: Original file is reconstructed and triggers a browser download

DATA FLOW DIAGRAM:
------------------
  [User's File]
       |
       v
  [AES-256-GCM Encrypt in Browser]
       |
       v
  [Encrypted Blob] --> [Upload to IPFS via Pinata]
       |                        |
       v                        v
  [Show CID]              [Stored on IPFS Network]
  [Show Password]               |
       |                        |
       v                        v
  [Receiver enters CID] --> [Fetch from IPFS Gateway]
  [Receiver enters KEY]         |
       |                        v
       v                   [Encrypted Blob Downloaded]
  [AES-256-GCM Decrypt in Browser]
       |
       v
  [Original File Downloaded]

4. ENCRYPTION DEEP DIVE
================================================================================

ALGORITHM: AES-256-GCM (Advanced Encryption Standard, Galois/Counter Mode)
---------------------------------------------------------------------------

  Property          | Value
  ------------------|------------------------------------------
  Algorithm         | AES (Advanced Encryption Standard)
  Mode              | GCM (Galois/Counter Mode)
  Key Size          | 256 bits (32 bytes)
  IV Size           | 96 bits (12 bytes) — NIST SP 800-38D
  Auth Tag          | 128 bits (16 bytes) — built into GCM
  Library           | window.crypto.subtle (Web Crypto API)
  External Libs     | NONE — zero dependencies

WHY AES-256-GCM?
  - AES-256 is military-grade. Used by governments, banks, intelligence
    agencies worldwide. Brute-forcing a 256-bit key would take billions
    of years with current technology.

  - GCM mode provides BOTH:
    a) Confidentiality — nobody can read the data
    b) Authentication — nobody can tamper with it
    If even a single bit of encrypted data is modified, decryption fails.

  - Web Crypto API is:
    a) Hardware-accelerated (uses CPU AES-NI instructions)
    b) Maintained by browser vendors (Chrome, Firefox, Safari, Edge)
    c) Not vulnerable to JavaScript timing attacks
    d) FIPS 140-2 compliant in most browsers

KEY GENERATION:
  - Uses crypto.subtle.generateKey() with 256-bit length
  - New key generated PER FILE — never reused
  - Key is exported as Base64 for sharing

IV (INITIALIZATION VECTOR):
  - Uses crypto.getRandomValues() for true randomness
  - 96 bits as recommended by NIST SP 800-38D
  - New IV generated PER ENCRYPTION — even same file encrypted twice
    produces completely different ciphertext

PACKET FORMAT (stored on IPFS):
  [2 bytes: metadata length (uint16 LE)]
  [N bytes: JSON metadata {"n":"filename","t":"mimetype"}]
  [12 bytes: Initialization Vector]
  [Remaining: AES-GCM ciphertext + 16-byte auth tag]

5. THE DUAL CREDENTIAL SYSTEM (CID + PASSWORD)
================================================================================

BurnerDrop uses a "double encryption" philosophy where file access requires
TWO independent pieces of information:

  CREDENTIAL 1: CID (Content Identifier)
  ----------------------------------------
    - IPFS content-addressed hash (e.g., bafkreig7hk5...)
    - Tells the system WHERE the encrypted blob is stored
    - Generated by IPFS based on the content hash of the encrypted data
    - Unique per file — content-addressed, not location-addressed

  CREDENTIAL 2: PASSWORD (AES-256 Key)
  ----------------------------------------
    - The actual decryption key formatted as human-readable segments
    - Example: aX4k-mR9q-bL2w-pN7j-kT8x-vD3m-qR5n-wF9s
    - 256 bits of entropy (cryptographically secure)
    - Generated by window.crypto.subtle

SECURITY MATRIX:
  Attacker has ONLY CID    -> Can download blob, but it's gibberish without key
  Attacker has ONLY KEY    -> Has the key but can't find the file on IPFS
  Attacker has BOTH        -> Can decrypt (that's why: NEVER share together)

RECOMMENDED SHARING:
  - Send CID via one channel (e.g., email)
  - Send Password via another channel (e.g., WhatsApp, Signal, in person)
  - This way, even if one channel is compromised, the file is still safe

6. IPFS & DECENTRALIZED STORAGE
================================================================================

WHAT IS IPFS?
  InterPlanetary File System — a decentralized, peer-to-peer storage network.

  Traditional Storage:        IPFS:
  "Get file from server X"    "Get file with hash Y" (content-addressed)
  Single point of failure     Distributed across nodes
  Can be censored/seized      Censorship-resistant
  Location-based              Content-based

HOW BURNERDROP USES IPFS:
  1. Encrypted blob uploaded to Pinata (IPFS pinning service)
  2. Pinata ensures the blob is available on the IPFS network
  3. CID (content hash) returned — this is the file's address
  4. Receivers fetch via public IPFS gateway

PINATA INTEGRATION:
  - API: https://api.pinata.cloud/pinning/pinFileToIPFS
  - Auth: JWT bearer token (entered by user in config panel)
  - Gateway: https://gateway.pinata.cloud/ipfs/{CID}
  - CID Version: v1 (modern, compact format)

WHAT PINATA/IPFS SEES:
  - Encrypted binary blob (unintelligible)
  - Upload timestamp
  - Blob size
  - That's it. No filename, no file type, no key, no user identity.

7. SECURITY ANALYSIS
================================================================================

WHAT THE SERVER NEVER SEES:
  [x] Original file content
  [x] File name or type
  [x] Encryption key
  [x] Initialization vector (embedded in blob)
  [x] Who sent or received the file
  [x] IP correlation (IPFS is peer-to-peer)

SECURITY MEASURES IMPLEMENTED:
  [+] AES-256-GCM encryption (military-grade)
  [+] Fresh key per file (no key reuse)
  [+] Fresh IV per encryption (no IV reuse)
  [+] GCM authentication tag (tamper detection)
  [+] Content Security Policy (CSP) headers
  [+] X-Frame-Options: DENY (anti-clickjacking)
  [+] Referrer-Policy: no-referrer (no URL leaking)
  [+] Rate limiting on decrypt attempts (5 max, 30s cooldown)
  [+] API key stored in sessionStorage only (cleared on tab close)
  [+] File size validation (10 MB limit)
  [+] No external JavaScript dependencies (zero supply chain risk)

AUTOMATED TEST SUITE (10 TESTS, ALL PASSING):
  Test 1: Key Generation — AES-256-GCM key + Base64 export
  Test 2: Text File Roundtrip — encrypt -> decrypt -> content match
  Test 3: Binary File Roundtrip — 4 KB binary, byte-for-byte match
  Test 4: Wrong Key Rejection — wrong key -> GCM auth tag fails
  Test 5: File Size Limit — >10 MB -> error thrown
  Test 6: Unique IV — same file encrypted 3x -> 3 different IVs
  Test 7: Share Link Roundtrip — build URL -> parse -> all fields match
  Test 8: Multiple MIME Types — HTML, JSON, CSV all roundtrip
  Test 9: API Key Guard — upload without JWT -> error thrown
  Test 10: Invalid Link Rejection — malformed link -> error thrown

KNOWN SECURITY GAPS (to fix before production):
  - Pinata JWT exposed in frontend (needs backend proxy)
  - No automated file deletion/expiry mechanism
  - Plaintext not zeroed from browser memory after encryption
  - No Subresource Integrity on Google Fonts CDN
  - No HTTPS enforcement (must be handled by hosting provider)

9. COMPETITOR COMPARISON: BURNERDROP VS. BITWARDEN SEND
================================================================================

Bitwarden Send is currently one of the most popular secure text/file sharing
tools available, built into the Bitwarden Password Manager. While they share
a common goal (secure sharing), their architectural approaches represent
different philosophies.

FEATURE COMPARISON MATRIX:
  Feature              | BurnerDrop             | Bitwarden Send
  ---------------------|------------------------|---------------------------
  Sender Account Needs | NO (Anonymous)         | YES (Must have BW account)
  Receiver Login Needed| NO                     | NO
  File Sharing (Free)  | YES (10 MB)            | NO (Premium required)
  Max File Size        | 10 MB (Free)           | 500 MB (Paid only)
  Encryption           | E2E (AES-256-GCM)      | E2E (AES-256-CBC)
  Storage Mechanism    | Decentralized (IPFS)   | Centralized (Azure Cloud)
  Application Surface  | Browser-native API     | Desktop/Mobile/Web/Ext
  Dual Credentials     | YES (CID + Password)   | NO (Single link + optional pw)
  Self-Destruct/Expiry | No (Persistent on IPFS)| YES (Timed / Read-Count limit)
  Identity Correlation | Impossible             | Correlated to Sender's BW Acct
  External Dependencies| Zero                   | Many

WHY CHOOSE BURNERDROP OVER BITWARDEN SEND?
  1. True Anonymity: Bitwarden Send requires the sender to log into their
     personal or corporate Bitwarden vault. The "Send" is permanently
     correlated to that user's identity. BurnerDrop requires no login —
     it is mathematically un-linkable to the sender.
  2. Free File Sharing: Bitwarden restricts file attachments entirely to
     their paid Premium tier (free users can only send text). BurnerDrop
     handles binary files natively for free.
  3. Decentralized Storage: If Bitwarden's centralized cloud goes down
     (or is compelled by authorities to shut down a link), the file is
     gone. BurnerDrop uses IPFS; once pinned, the file is censorship-
     resistant and unaffected by single-server outages.
  4. Independent Credentials: BW Send utilizes a single URL. An optional
     password can be added. BurnerDrop intrinsically enforces a dual-
     credential model (CID + Password) by design.

WHY CHOOSE BITWARDEN SEND OVER BURNERDROP?
  1. File Expiration: The biggest advantage of BW Send is auto-deletion.
     You can set a file to vanish after 1 hour or 5 views. Because
     BurnerDrop uses IPFS, content is notoriously difficult to un-pin or
     guarantee actual deletion across the decentralized network.
  2. File Limits: BW Send allows up to 500 MB per file (for paid users).
     BurnerDrop is currently constrained to 10 MB due to public IPFS gateway
     and browser memory limits.
  3. Ecosystem Convenience: If you already use Bitwarden, creating a Send
     is incredibly fast via their browser extension without needing to
     navigate to a separate website.

10. COMPETITOR COMPARISON: BURNERDROP VS. ONIONSHARE
================================================================================

OnionShare is a highly respected privacy tool developed by Micah Lee and backed
by the Tor Project. It allows users to securely and anonymously share files,
host websites, and chat using the Tor network. Both BurnerDrop and OnionShare
target the same privacy-conscious individual, but they use fundamentally
different architectures.

FEATURE COMPARISON MATRIX:
  Feature              | BurnerDrop             | OnionShare
  ---------------------|------------------------|---------------------------
  Delivery Network     | IPFS (Clearweb HTTP)   | The Tor Network (.onion)
  Application Format   | Browser Web App        | Desktop App (Python/Qt)
  Uptime Requirement   | None (Upload & leave)  | Sender must remain online
  Sender Anonymity     | High (No accounts)     | Maximum (Tor routing)
  Setup Friction       | Zero (Load URL)        | High (Download/Install Tor)
  Storage Mechanism    | Decentralized Nodes    | Sender's Local Hard Drive
  Receiver Setup       | Standard Browser       | Tor Browser Required
  Max File Size        | 10 MB limit            | Unlimited (Disk Capacity)

WHY CHOOSE BURNERDROP OVER ONIONSHARE?
  1. Zero Friction / Convenience: BurnerDrop is a regular website you can
     open in Chrome/Safari/Edge. OnionShare requires the sender to download
     and install a dedicated desktop application, and it requires the 
     receiver to download and install the Tor Browser.
  2. Asynchronous Sharing: With BurnerDrop, you encrypt the file, upload it
     to IPFS, get the CID+Password, and you can close your laptop. The receiver
     can fetch it 3 days later. With OnionShare, your computer IS the server.
     If you close your laptop or the app before the receiver downloads the
     file, the transfer fails.
  3. No Infrastructure Knowledge Needed: BurnerDrop feels like a modern SaaS
     app. OnionShare can feel technical and intimidating to non-technical users
     who don't understand how Tor onion routing works.

WHY CHOOSE ONIONSHARE OVER BURNERDROP?
  1. Maximum Traffic Anonymity: OnionShare uses Tor hidden services. This
     means ISP snoopers cannot even see *that* you are sharing a file, let
     alone what the file is. With BurnerDrop, your ISP can see you are talking
     to a public IPFS gateway like Pinata.
  2. Unlimited File Sizes: Because OnionShare serves the file directly from
     your hard drive, the only limit is the size of the receiver's hard drive 
     and internet speed. You can easily share a 50 GB database dump. BurnerDrop
     caps out quickly via browser memory and IPFS gateway limits (10 MB).
  3. True Ephemeral Transfer: Once you stop sharing in OnionShare, the file
     is perfectly gone, because it was only ever on your laptop. BurnerDrop
     content is pinned to IPFS, meaning it persists on public nodes until
     it falls out of the pinning cache.

GENERAL COMPETITOR COMPARISON (OTHERS):

  Feature              | BurnerDrop | WeTransfer | ProtonDrive | OnionShare
  ---------------------|------------|------------|-------------|----------
  No Account Required  | Yes        | Yes*       | No          | Yes
  E2E Encrypted        | Yes        | No         | Yes         | Yes
  Zero Knowledge       | Yes        | No         | Yes*        | Yes
  Client-Side Crypto   | Yes        | No         | No          | Yes
  Decentralized Storage| Yes (IPFS) | No         | No          | Yes (Tor)
  No Tracking          | Yes        | No         | Minimal     | Yes
  Open Source          | Yes        | No         | Partial     | Yes

  * WeTransfer has tracking, analytics, ads in free tier
  * ProtonDrive controls the server, so "zero knowledge" has caveats

UNIQUE DIFFERENTIATORS:
  1. Dual Credential System — no other tool requires CID + separate password
  2. IPFS Storage — decentralized, censorship-resistant, content-addressed
  3. Zero Dependencies — entire crypto stack is browser-native
  4. ~75 KB Total — entire app is smaller than a single React component
  5. No Build Step — clone and serve, that's it

12. STRENGTHS (WHAT MAKES BURNERDROP DIFFERENT)
================================================================================

  1. TRUE ZERO KNOWLEDGE
     Not "we promise we don't look" — mathematically impossible.
     The server never has the key. Period.

  2. ZERO DEPENDENCIES
     No npm. No webpack. No React. No node_modules.
     The entire crypto stack is 1 file using browser-native APIs.
     This means: zero supply chain attacks, zero dependency vulnerabilities.

  3. DUAL CREDENTIALS
     Every other tool gives you ONE link/code. BurnerDrop gives you TWO
     independent credentials. Even if one leaks, your file stays safe.
     This is a genuine security innovation for consumer file sharing.

  4. IPFS STORAGE
     Decentralized = no single point of failure, censorship-resistant,
     content-addressed (cannot be silently modified).

  5. RADICAL SIMPLICITY
     75 KB total codebase. No build step. No server-side code.
     Clone → serve → done. Any developer can audit the entire app
     in under an hour.

  6. PRIVACY BY DESIGN
     No accounts, no cookies, no analytics, no tracking pixels.
     The app literally cannot identify its users.

13. WEAKNESSES & RISKS
================================================================================

TECHNICAL:
  - 10 MB file limit (IPFS gateway constraints)
  - No file expiry mechanism (IPFS content is persistent)
  - Pinata API key exposed in frontend code (needs backend proxy)
  - Browser memory not zeroed after encryption
  - No offline support (requires internet for IPFS)
  - Large files could cause browser memory issues (no streaming encryption)

BUSINESS:
  - Hard to monetize a privacy tool (privacy-conscious users resist tracking)
  - IPFS pinning costs money (~$0.15/GB/month on Pinata)
  - No user accounts = no retention metrics, no engagement tracking
  - Competing with free tools (WeTransfer, Google Drive)
  - Educating users about CID + Password is a UX friction point

LEGAL:
  - E2E encryption is legally restricted in some countries
  - Could be used for illegal file sharing (liability concerns)
  - IPFS content persistence conflicts with "right to be forgotten" (GDPR)
  - No DMCA takedown mechanism for IPFS-hosted content

MARKET:
  - Consumer privacy tools have historically struggled to monetize
    (see: Firefox Send shutdown, Keybase acquisition)
  - Enterprise customers may want centralized control (conflicts with
    zero-trust architecture)
  - Network effect is weak (each transfer is isolated, no social graph)

  - Need dedicated IPFS pinning (Pinata Business: $100/month)
  - Consider running own IPFS cluster (3 nodes: ~$150/month on AWS)
  - Add Cloudflare CDN for static assets
  - Estimated cost: $200-400/month

AT 100,000 USERS/DAY:
  - Self-hosted IPFS cluster (10+ nodes)
  - Multiple gateway endpoints
  - Geographic distribution
  - Backend API for premium features
  - Estimated cost: $2,000-5,000/month

KEY ADVANTAGE:
  Because all crypto is client-side, server costs scale linearly with
  STORAGE, not with COMPUTE. This is 10-100x cheaper than server-side
  encryption at scale.

17. CONCLUSION
================================================================================

BurnerDrop is a technically sound, privacy-first file sharing application
that addresses a genuine market gap. Its key innovations — dual credential
system, zero dependencies, and IPFS storage — differentiate it from both
mainstream tools (WeTransfer, Google Drive) and niche privacy tools
(OnionShare, ProtonDrive).

The project's greatest strength is its radical simplicity: 75 KB of
auditable code, zero external dependencies, and a zero-trust architecture
that makes security guarantees mathematical rather than promissory.

The primary challenges are:
  1. Monetization (privacy tools are hard to charge for)
  2. UX education (dual credentials add friction)
  3. File size limits (IPFS gateway constraints)
  4. IPFS content persistence (no auto-delete)

The recommended path forward is an Open Core model: keep the current app
free and open source (building trust and adoption), while monetizing
